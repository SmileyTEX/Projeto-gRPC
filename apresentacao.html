Projeto gRPC · Python

Gacha Game

Sistema distribuído com comunicação remota via gRPC e contrato em Protocol Buffers.

Escopo

O que construímos

Entidade

  • Personagem
  • nome, raridade, elemento, nível

Operações remotas

  • InvocarPersonagem
  • ListarPersonagens
  • EvoluirPersonagem

Pergunta 1

Por que o contrato foi definido assim?

  • Uma mensagem composta (Personagem) concentra os atributos do domínio.
  • RPCs com nomes do jogo deixam a intenção clara: invocar, listar, evoluir.
  • Request/Response separados facilitam evoluir o serviço sem quebrar o cliente.
  • Contrato único (.proto) compartilhado por cliente e servidor reduz inconsistências.

Contrato

Exemplo do .proto

service GachaService { rpc InvocarPersonagem (InvocarRequest) returns (InvocarResponse); rpc ListarPersonagens (ListarRequest) returns (ListarResponse); rpc EvoluirPersonagem (EvoluirRequest) returns (EvoluirResponse); } message Personagem { int32 id = 1; string nome = 2; string raridade = 3; string elemento = 4; int32 nivel = 5; }

Pergunta 2

Como ocorre a comunicação?

Cliente
→
Canal gRPC
→
Servidor
→
Resposta Protobuf
  • Cliente abre canal para localhost:50051.
  • Usa o stub gerado como se chamasse uma função local.
  • Mensagem é serializada em Protobuf e enviada ao servidor.
  • Servidor processa (memória) e devolve uma resposta tipada.

Arquitetura

Papéis no sistema

server.py

  • Implementa o serviço
  • Mantém a coleção em memória
  • Aplica regras (sorteio, evolução)

client.py

  • Menu por linha de comando
  • Chama os RPCs remotos
  • Exibe as respostas ao usuário

Pergunta 3

Arquivos gerados automaticamente

gacha_pb2.py Classes das mensagens: Personagem, requests e responses.
gacha_pb2_grpc.py Stubs do cliente e base do servidor (GachaServiceStub / Servicer).

Gerados a partir de gacha.proto com python generate_stubs.py. Não se edita à mão.

Pergunta 4

gRPC × REST neste cenário

API REST

  • Rotas HTTP + JSON
  • Contrato mais informal
  • Bom para APIs públicas na web

gRPC (nossa escolha)

  • Contrato tipado no .proto
  • Código gerado automaticamente
  • Chamadas RPC explícitas
  • Serialização binária eficiente

Vantagens

Por que gRPC faz sentido aqui?

  • Serviço interno entre processos — não precisamos de URLs REST para cada ação.
  • Operações com nomes claros — InvocarPersonagem é mais direto que POST /personagens.
  • Menos boilerplate — stubs gerados evitam serializar/deserializar JSON à mão.
  • Contrato versionável — mudanças no domínio partem do .proto.

Encerramento

Resumo

  • Contrato centrado na entidade Personagem e 3 RPCs do gacha.
  • Cliente e servidor se comunicam por canal gRPC + Protobuf.
  • Arquivos gerados: gacha_pb2.py e gacha_pb2_grpc.py.
  • gRPC traz tipagem, geração de código e eficiência para este cenário.
